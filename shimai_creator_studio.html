Podcast ops cockpit · RSS + GLM

把播客生产做成一套可运营的专业工作台

这个原型页不是单纯的官网补充，而是围绕时髦小姨主播运营的真实工作流搭建：从 RSS 音频流进来，到 GLM 语义分析、选题 Brainstorm、逐字稿写作、录制建议、剪辑建议，再到发布工作流和复盘闭环。

看工作流 看页面结构 看目录蓝图
5核心动作：选题、写稿、录制、剪辑、发布
1个内容中台：RSS + 语义分析 + 任务流
MVP先落地 Dashboard / Brainstorm / Scripts / Publish
GLM作为主分析模型，保留可回放与可追踪日志

本页先验证什么

  • 1运营能不能在首页直接看见“今天该做什么”。
  • 2选题建议能不能解释清楚“为什么是这个方向”。
  • 3逐字稿和录制/剪辑建议能不能直接交给协作人用。
  • 4发布工作流能不能形成状态机，而不是松散待办。

内容生产流水线

这条链路把播客从“内容”变成“可管理的生产系统”。每一步都应该输出结构化结果，而不是只给一段聊天式回答。

01 RSS Sync

接入现有播客音频流，拉取节目标题、简介、发布时间、音频链接与历史状态。

02 GLM Analyze

做语义分析、主题簇、情绪、人物、冲突点和可延展母题提取。

03 Brainstorm

把历史节目和趋势素材转成选题卡，显示评分、相似内容与风险提示。

04 Script / Record / Edit

逐字稿、录制建议、剪辑建议分三层输出，让协作人拿到可以直接执行的材料。

05 Publish

生成发布检查清单、渠道分发内容与复盘记录，形成稳定工作流闭环。

页面骨架

原型先按页面切分，而不是按模型能力切分。这样产品、设计、前后端都能直接认领。

Dashboard 总览

首页先回答三个问题：今天做什么、有什么风险、模型刚刚产出了什么。

  • 今日待办
  • 最新节目与同步状态
  • 模型摘要和风险提示

选题 Brainstorm

用主题卡、评分、相似节目和机会点，把“灵感”变成可讨论的候选集。

  • 选题推荐卡
  • 受众标签与相似内容
  • 风险点与可讲述性评分

逐字稿工坊

把选题直接转成可编辑文本，支持版本、批注、语气切换和段落重写。

  • 大纲 + 正文编辑区
  • 版本对比与批注
  • 金句和转场建议

录制建议

输出录制平台、设备、环境和节奏建议，避免创作和执行之间断层。

  • 录制平台选择
  • 设备与环境清单
  • 录制前 checklist

剪辑建议

把内容切点、章节点、标题和封面摘要整理成剪辑师能直接用的输出。

  • 时间轴建议
  • 停顿与切点
  • 标题 / 封面 / 摘要

发布工作流

把发布页做成状态机，而不是一个“发完就算了”的提交按钮。

  • 待确认 / 待录制 / 待剪辑
  • 待审核 / 待发布 / 已发布
  • 复盘记录与责任人

后台模块

这个原型先把后台拆成 4 个最先验证价值的区域，避免一开始做成“菜单很多但没人用”的空壳。

内容资产库

历史节目、主题簇、语义标签、标题和封面素材沉淀区。

ArchiveTagsReuse

系统设置

RSS、GLM、权限、工作流规则和发布策略集中配置。

ConfigAccessRules

模型运行

保存每次调用的输入摘要、输出版本、耗时和错误信息。

TraceLogsReplay

协作评论

把“模型建议”变成可讨论、可修改、可确认的协作产物。

ReviewEditApprove

信息架构原则

默认优先展示结论和可执行建议，再下钻到证据和原始分析。运营不应该先看到模型日志。

先总览Dashboard
再判断Brainstorm
再写作Scripts
再执行Recording / Editing
再发布Publish

推荐技术栈

先用能快速迭代的组合把闭环跑起来，再在向量检索、主题聚类和自动化发布上逐步增强。
