# 时髦小姨创作者专业站 PRD & 架构草案

## 1. 需求摘要

目标不是再做一个普通播客官网，而是把时髦小姨的内容生产流程做成一个主播运营可直接使用的专业工作台。核心链路是：RSS 音频流接入 -> 音频语义分析 -> 选题推荐 -> 逐字稿编写 -> 录制建议 -> 剪辑建议 -> 发布工作流 -> 复盘与再推荐。

现有站点已经有播客前台、节目页和基础内容结构，因此新站更适合升级为“前台官网 + 后台创作控制台 + 语义分析引擎”的组合，而不是推倒重来。

## 2. 业务场景与使用目的

主要使用者是时髦小姨主播运营、内容编导和剪辑协作人。典型场景包括：

- 每周从 RSS 与历史节目中找出可继续延展的话题。
- 把一个选题快速扩写成适合录制的逐字稿。
- 录制前给出平台、设备、麦克风、环境和节奏建议。
- 剪辑阶段输出结构化剪辑点、封面文案、标题与摘要。
- 发布前自动生成元数据、分发清单和复盘任务。

## 3. 目标用户与角色分层

- 主播：查看选题、编辑逐字稿、确认最终发布。
- 运营：管理选题池、安排发布节奏、观察播客表现。
- 编导：基于语义分析和历史内容做深度选题与脚本结构设计。
- 剪辑：读取录音建议、时间轴、切点建议和素材标注。
- 管理员：维护 RSS 源、模型配置、权限和工作流规则。

## 4. 用户旅程与关键任务流

1. 系统自动拉取 RSS 播客音频流，入库节目元数据与摘要。
2. GLM 对音频文本、摘要、标题、关键词做语义分析，生成主题簇、情绪、人物、冲突点和可延展母题。
3. 运营在“选题 Brainstorm”页看到推荐选题、理由、风险和受众匹配度。
4. 选择一个选题后进入“逐字稿工坊”，由模型生成提纲、分段、开场、转场、金句和收尾。
5. 进入“录制建议”页，输出录制平台、设备、空间、镜头、节奏与避坑建议。
6. 进入“剪辑建议”页，输出剪辑结构、停顿处理、B-roll/音效建议、标题与封面建议。
7. 发布页生成分发清单、社媒摘要、RSS 更新说明与待办。

## 5. 功能范围

### MVP
- RSS 源配置与节目同步。
- 节目库、选题池、逐字稿编辑器。
- GLM 语义分析摘要卡片。
- 录制建议与剪辑建议模板生成。
- 发布工作流页面。

### V1
- 选题评分、受众标签、历史内容重合度检测。
- 逐字稿版本管理与协作评论。
- 自动提炼标题、Shownotes、摘要、章节点。
- 录音/剪辑任务单导出。

### V2
- 趋势选题建议、跨集主题聚类、内容缺口洞察。
- 多模态分析：音频、转写、封面图、社媒数据联动。
- 自动化发布编排与通知。

### Out of Scope
- 复杂的多租户商业计费。
- 全自动无人审核发布。

## 6. 非功能需求

- 响应速度：常用查询 P95 < 2.5s。
- 分析任务：异步处理，允许分钟级延迟。
- 可用性：核心创作页面可在单点故障后恢复。
- 可观测性：每次分析与生成保留 prompt、输入摘要、输出版本和错误日志。
- 安全：模型密钥与 RSS token 只存在服务端，不下发前端。

## 7. 信息架构与核心页面

建议页面结构：

- 首页 Dashboard：今日待办、最新节目、内容状态、待审任务。
- 选题 Brainstorm：主题推荐、趋势、相似节目、机会点、风险点。
- 逐字稿工坊：大纲、逐段写作、版本对比、批注。
- 录制建议：平台推荐、录制检查清单、设备与环境建议。
- 剪辑建议：时间轴建议、切点、金句、封面、标题、摘要。
- 发布工作流：RSS 更新、平台分发、发布检查、复盘。
- 内容资产库：历史节目、主题簇、语义标签、素材库。
- 系统设置：RSS、GLM、权限、工作流规则。

## 8. 系统架构与前后端框架建议

推荐采用“模块化单体 + 异步任务层”的方案，而不是一开始拆微服务。

### 方案 A：保守方案
- 前端：React / Next.js。
- 后端：FastAPI。
- 数据库：PostgreSQL。
- 缓存与队列：Redis。
- 分析任务：Celery / RQ / 自定义 worker。

优点：实现快，适合现阶段功能集中、需求还在快速迭代的状态。

### 方案 B：进取方案
- 前端：Next.js + App Router。
- 后端：FastAPI + 异步任务编排层。
- 数据库：PostgreSQL + pgvector。
- 搜索：Postgres FTS 或独立搜索引擎。
- 可观测性：OpenTelemetry + 结构化日志。

优点：更适合后续做语义检索、内容推荐和主题聚类。

### 推荐结论
优先选方案 B，但第一阶段只启用最小必要子集。原因是你明确要做“音频语义分析”和“后续选题建议”，这类能力很快会依赖向量检索、任务编排和分析回放能力，直接按可演进结构设计更省返工。

## 9. 技术栈建议

- 前端语言：TypeScript。
- 前端框架：Next.js。
- UI：Tailwind CSS + 组件库。
- 后端语言：Python。
- 后端框架：FastAPI。
- 任务队列：Redis + Celery 或 RQ。
- 数据库：PostgreSQL。
- 语义向量：pgvector。
- 模型接入：GLM API 作为主模型，预留多模型切换层。
- 文件存储：对象存储或本地静态存储，后续可扩展到 S3 类方案。

Python 更适合你现在已有的 RSS 同步、音频处理、语义分析脚本生态；前端用 TypeScript 更适合做一个专业、交互密集的创作者后台。

## 10. 数据模型与数据库 schema 初稿

核心表建议：

- `podcast_feeds`：RSS 源配置。
- `episodes`：节目主表，保存标题、发布时间、音频 URL、转写状态。
- `episode_assets`：音频、封面、转写、摘要等资产。
- `semantic_tags`：主题、情绪、人物、地点、冲突点标签。
- `topic_candidates`：选题候选与评分。
- `draft_scripts`：逐字稿草稿与版本。
- `recording_guides`：录制建议输出。
- `editing_guides`：剪辑建议输出。
- `publish_runs`：发布工作流记录。
- `model_runs`：GLM 调用日志与结果摘要。

建议字段统一带审计列：`created_at`、`updated_at`、`created_by`、`updated_by`。

示例主键与关系：

- `episodes.feed_id -> podcast_feeds.id`
- `topic_candidates.episode_id -> episodes.id` 或 `topic_candidates.cluster_id`
- `draft_scripts.topic_candidate_id -> topic_candidates.id`
- `model_runs.target_type / target_id` 作为通用引用，方便追踪每次分析对象

## 11. API 合同草案

建议 REST 优先。

- `GET /api/feeds`：列出 RSS 源。
- `POST /api/feeds/sync`：触发 RSS 同步。
- `GET /api/episodes`：节目列表。
- `GET /api/episodes/{id}`：节目详情。
- `POST /api/topics/brainstorm`：生成选题建议。
- `POST /api/scripts/draft`：根据选题生成逐字稿。
- `POST /api/recording-guides/generate`：生成录制建议。
- `POST /api/editing-guides/generate`：生成剪辑建议。
- `POST /api/publish-runs`：创建发布工作流。
- `GET /api/model-runs/{id}`：查看模型调用与结果。

错误码建议：

- `400` 参数错误。
- `401` 未认证。
- `403` 无权限。
- `404` 资源不存在。
- `409` 版本冲突或重复生成。
- `429` 频率限制。
- `500` 服务端异常。

## 12. 指标体系与埋点需求

业务指标：

- 每周可用选题数。
- 选题到逐字稿的平均耗时。
- 逐字稿修改轮次。
- 发布前检查通过率。
- 节目发布后被复用到社媒/简介的次数。

技术指标：

- RSS 同步成功率。
- GLM 调用成功率与平均延迟。
- 任务队列积压长度。
- 语义分析错误率。

埋点建议覆盖：页面访问、选题采纳、逐字稿生成、手工改写、发布提交、模型重试、导出下载。

## 13. 安全、权限与合规要求

- 管理员才能修改 RSS、模型配置与发布规则。
- 逐字稿与模型输出按节目版本留痕。
- 所有外部 API Key 服务端托管。
- 如果后续接入未公开音频内容，需要考虑版权与存储期限策略。
- 如果录音包含个人信息或未发布素材，需要加访问控制和审计日志。

## 14. 风险清单与约束条件

- 风险 1：语义分析质量不足，导致选题建议泛化。
- 缓解：先做“可解释推荐”，展示命中依据、相似节目和主题簇。

- 风险 2：逐字稿生成过长或过于 AI 化。
- 缓解：把输出拆成提纲、口语化正文、金句与转场四层。

- 风险 3：发布工作流和实际平台不一致。
- 缓解：把工作流定义为“内部审核与导出标准”，不要一开始假设全自动发布。

- 风险 4：RSS 源不稳定或字段不统一。
- 缓解：加状态机、重试、去重与手工校正入口。

## 15. 里程碑计划与资源估算

### Phase 0：原型与规则冻结
- 明确页面结构、字段、工作流和输出模板。
- 产出可点击原型或静态页面骨架。

### Phase 1：MVP
- RSS 同步。
- 节目库。
- 选题 Brainstorm。
- 逐字稿生成。
- 录制建议。
- 剪辑建议。
- 发布工作流。

### Phase 2：增强
- 向量检索。
- 主题聚类。
- 历史内容复用推荐。
- 版本对比与协作评论。

### 资源估算
- 1 名前端。
- 1 名后端/数据工程。
- 0.5 名内容运营参与标注与验收。

## 16. 验收标准

- 能成功接入至少 1 个 RSS 源并同步节目元数据。
- 能基于历史节目生成至少 3 条有解释的选题建议。
- 能从一个选题生成可编辑的逐字稿草案。
- 能输出录制平台和剪辑建议。
- 能生成发布检查清单与导出结果。
- 所有模型结果都能回溯到输入和版本。

## 17. 待确认问题清单

- RSS 源地址是否已经确定，是否只有一个源还是多个源？
- GLM 具体使用哪个版本和计费方式？
- 是否需要支持人工转写导入，还是仅依赖音频自动分析？
- 逐字稿是否要求支持多轮协作评论？
- 发布阶段是否要对接具体平台，还是先只做内部发布工作流？
- 是否需要中英双语输出？
- 是否要沿用当前时髦小姨官网视觉风格，还是做全新后台风格？

## 推荐落地顺序

1. 先做创作者后台的首页、选题页、逐字稿页和发布页四个核心页面。
2. 再接 RSS 同步和节目库。
3. 再接 GLM 语义分析与选题推荐。
4. 最后补向量检索、主题聚类和自动化发布。

## 结论

这个项目最合理的形态不是“单纯播客站”，而是“内容生产操作台 + 播客前台”。前台继续承接品牌与收听，后台承接选题、写作、录制、剪辑和发布。技术上建议用 Python 后端承接 RSS 和 GLM，前端用 TypeScript 做专业工作台，这样可以沿着你已有的内容资产逐步升级，而不是重建一套孤岛系统。
